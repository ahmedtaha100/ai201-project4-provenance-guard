# 🛡️ Provenance Guard

A Flask backend any creative-sharing platform can plug into to answer one
question about a piece of writing — *was this written by a human, or generated by
an AI?* — and to answer it **honestly**: with calibrated, uncertainty-aware
confidence, a plain-language transparency label, and a creator appeal path, behind
rate limiting and a structured audit log.

> CodePath **AI201 — Applications of AI Engineering**, Project 4 (Show What You
> Know). Design doc: [`planning.md`](planning.md).

The hard part of AI detection is not getting a number — it is being *honest* about
an unsolved problem. Provenance Guard is built around that: it refuses to flip a
coin at 0.5, it makes wrongly accusing a human the *harder* call to reach, and it
gives creators a way to push back.

## Quick start

```bash
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash);  .venv\Scripts\activate on cmd
# source .venv/bin/activate         # macOS / Linux
pip install -r requirements.txt

cp .env.example .env                # then put your Groq key in .env
# GROQ_API_KEY=gsk_...               (free key: https://console.groq.com/keys)

python app.py                       # serves on http://localhost:5000
```

Then open **http://localhost:5000** for the web UI — paste text, see the
attribution, confidence, per-signal scores, and label, and appeal a result. Or
use the API directly:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "ok so i finally tried that ramen place downtown and honestly? underwhelming.", "creator_id": "alice"}'
```

> **Running without a Groq key.** If `GROQ_API_KEY` is unset, the LLM signal
> transparently falls back to a deterministic lexical heuristic so the whole
> pipeline still runs end-to-end (every response is flagged `"fallback_used": true`).
> A live key is required for *real* semantic classification — the fallback is for
> testing and outage resilience only.

## Architecture overview

A submission's path from raw text to the label a reader sees:

```
POST /submit {text, creator_id, content_type?}
      │ validate (non-empty, ≤50k chars, known type)
      ▼
  ┌── Signal 1: LLM semantic (Groq) ──┐  ┌ Signal 2: stylometric ┐  ┌ Signal 3: repetition ┐
  │   p_llm ∈ [0,1]                   │  │ p_style ∈ [0,1]        │  │ p_rep ∈ [0,1] (≥50w)  │
  └───────────────┬───────────────────┘  └───────────┬───────────┘  └──────────┬───────────┘
                  └─────────────► scoring.combine ◄────────────────────────────┘
                          weighted blend → pull toward 0.5 on disagreement
                          → asymmetric bands  →  confidence + attribution
                  ▼
          labels.make_label  →  1 of 3 plain-language variants  (+ Verified-Human badge?)
                  ▼
          audit.record (SQLite)  →  content_id, every signal score, status
                  ▼
   JSON: { content_id, attribution, confidence, label, signals, … }

Appeal:  POST /appeal {content_id, creator_reasoning}
         → look up content_id → status="under_review" + log reasoning → confirmation
         → GET /reviewer/queue surfaces it to a human
```

Full diagram + component narrative: [`planning.md` › Architecture](planning.md#architecture).

## Detection signals

The pipeline combines **three genuinely distinct signals** (two required + one for
the ensemble stretch). Each returns `ai_probability ∈ [0,1]` = P(text is AI).

| Signal | What it measures | Why it was chosen | What it misses (blind spot) |
|---|---|---|---|
| **1. LLM semantic** — Groq `llama-3.3-70b-versatile` ([`signals/llm_signal.py`](signals/llm_signal.py)) | *Meaning / voice.* Does the text *read* as AI when judged holistically — idea flow, hedging, generic "AI voice," and (crucially) whether it carries **specific substance** or interchangeable filler. | It is the only signal that understands meaning, and meaning is where modern AI text gives itself away (or vindicates a formal human). | Non-deterministic; the easiest signal to fool with lightly human-edited AI; needs a live API/key (covered by the fallback). |
| **2. Stylometric** — pure Python ([`signals/stylometric.py`](signals/stylometric.py)) | *Structure.* Sentence-length **burstiness** (CV), **informality** (contractions, `! ? ...`, slang, lowercase starts), and **lexical diversity** (MATTR). | It never reads meaning, so it *fails differently* from the LLM — a real independent cross-check, computable with zero dependencies. | **Flags formal human prose as AI** (it is uniform + clean); misreads repetitive poems; unstable on short text. |
| **3. Repetition / predictability** — pure Python ([`signals/ensemble.py`](signals/ensemble.py), stretch) | *Token-sequence predictability.* N-gram reuse + reliance on common high-frequency words — a cheap perplexity proxy. | A third orthogonal axis (not meaning, not sentence-shape). | Needs ≥50 words; refrains/recipes repeat by design. Dropped from scoring on short text rather than guessing. |

"Distinct" is load-bearing here: Signal 1 judges *substance*, Signal 2 judges
*sentence shape*, Signal 3 judges *sequence repetition*. They disagree in
informative ways — which is exactly what the scoring layer exploits.

## Confidence scoring

**`confidence` = P(content is AI-generated), 0..1.** We picked what the number
means to a *user* first, then built the math to hit it
([`scoring.py`](scoring.py), constants in [`config.py`](config.py)):

- **0.5 = "we genuinely don't know."** Signals split or weak → the system declines
  to attribute the work. This is the honest-uncertainty zone.
- → **1.0** = increasingly confident **AI**; → **0.0** = increasingly confident **human**.

Three steps:

1. **Weighted blend** over the signals that ran (LLM 0.50 · stylometric 0.30 ·
   repetition 0.20, renormalised; base pair without ensemble is 0.60 / 0.40).
2. **Disagreement → uncertainty.** `spread = max(pᵢ) − min(pᵢ)`; the score is
   pulled toward 0.5 by `0.5·spread`. Conflicting evidence reads as *uncertain*,
   never as a confident verdict.
3. **Asymmetric bands**: `≥ 0.70 → likely_ai`, `≤ 0.40 → likely_human`, else
   `uncertain`.

**The false-positive asymmetry is real, not decorative.** Two mechanisms implement
"calling a human's work AI is the costlier error":
- The bar to assert **AI** (0.70, i.e. 0.20 above 0.5) is *farther from 0.5* than
  the bar to call **human** (0.40, i.e. 0.10 below) — see `attribution_for()` in
  [`scoring.py`](scoring.py) and `AI_THRESHOLD`/`HUMAN_THRESHOLD` in
  [`config.py`](config.py). Borderline-high scores land in *uncertain*, not *AI*.
- The disagreement pull means a formal human text the LLM clears but stylometry
  flags gets pulled into *uncertain* instead of being asserted as AI.

### How we validated it's meaningful (real numbers)

Run against the spec's four starter inputs. These are **representative live-Groq
values** (`temperature=0`); the LLM score can vary by a few hundredths run-to-run,
which nudges the exact confidence but not the band. Running the **deterministic
fallback** (no key) produces different individual numbers that land in the same
attributions — so the table is illustrative of behaviour, not a fixed contract.

| Input | `llm` | `stylometric` | **confidence** | **attribution** |
|---|---|---|---|---|
| Clearly **AI** ("transformative paradigm shift…") | 0.92 | 0.64 | **0.77** | `likely_ai` |
| Clearly **human** (casual ramen review) | 0.05 | 0.11 | **0.13** | `likely_human` |
| Formal **human** (monetary-policy paragraph) | 0.42 | 0.74 | **0.535** | `uncertain` |
| Lightly-edited **AI** (remote-work paragraph) | 0.25 | 0.68 | **0.43** | `uncertain` |

**Two contrasting examples (one high, one lower) — the actual numbers:**

- **High confidence → `likely_ai`, confidence `0.77`.** The "paradigm shift"
  paragraph: the LLM scored it `0.92` (saturated with generic, content-free
  scaffolding) and stylometry `0.64` (uniform, formal). They *agree*, so there is
  little pull and the score clears the 0.70 AI bar.
- **Lower confidence → `likely_human`, confidence `0.13`.** The ramen review: LLM
  `0.05`, stylometry `0.11`, repetition `0.21` — all three agree it's human
  (contractions, slang, bursty rhythm, sensory specifics).

The gap (0.77 vs 0.13) is what shows the score is a real variable, not a constant.
And the two *uncertain* rows show the design's headline behaviour: a formal human
paragraph and a lightly-edited AI paragraph both land in the honest middle rather
than being confidently mislabeled. (Tests pin all of this — see
[`tests/test_pipeline.py`](tests/test_pipeline.py).)

## Transparency label

The label is the only thing a non-technical reader sees, so it is plain language
and the confidence is expressed as a percentage. There are **exactly three
variants**, chosen by the attribution band (so the label can never contradict the
audit log). `{pct}` = `round(confidence·100)`. Canonical strings live in
[`labels.py`](labels.py); here they are **verbatim**:

Each is quoted **verbatim** below as a single unbroken string (`{pct}` is the
integer AI-likelihood percentage substituted at runtime):

**`likely_ai` (high-confidence AI):**

```text
🤖 Likely AI-generated — Our automated check estimates a {pct}% likelihood that this text was produced mainly by an AI tool, which is above our high-confidence threshold. We're surfacing this for transparency. It is an estimate, not proof — if you wrote this yourself, you can appeal and a human will review it.
```

**`likely_human` (high-confidence human):**

```text
✍️ Likely human-written — Our automated check found writing patterns consistent with a human author (estimated {pct}% likelihood of AI generation — low). No strong AI-generation signals were detected. This is an automated estimate, not a guarantee of authorship.
```

**`uncertain`:**

```text
❓ Uncertain — We can't confidently tell whether this was written by a person or by an AI (estimated {pct}% likelihood of AI generation, inside our 'unsure' range). Our signals were weak or in conflict, so we are deliberately not labeling it either way. Treat it as unverified; the creator can add context through an appeal.
```

Example with `{pct}` filled (live, the AI paragraph above, 77%):

```text
🤖 Likely AI-generated — Our automated check estimates a 77% likelihood that this text was produced mainly by an AI tool, which is above our high-confidence threshold. We're surfacing this for transparency. It is an estimate, not proof — if you wrote this yourself, you can appeal and a human will review it.
```

A verified creator (see [stretch](#stretch-features)) gets a `✓ Verified Human
creator` badge appended — independent of the per-post analysis.

## Appeals workflow

A creator who believes a classification is wrong contests it via `POST /appeal`
with `{content_id, creator_reasoning}`. The system ([`appeals.py`](appeals.py)):
looks up the original decision (→ 404 if unknown), stores the reasoning + timestamp
**beside** it (original verdict preserved), flips `status` → `under_review`, and
returns a confirmation. **No automated re-classification** — a human reviewer is
the deliberate safety valve. `GET /reviewer/queue` is what that reviewer sees.

```bash
curl -s -X POST http://localhost:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id": "51e7f1a3-...", "creator_reasoning": "I wrote this myself for a class."}'
# -> {"status": "under_review", "original_attribution": "likely_ai", "message": "...logged..."}
```

## Rate limiting

`POST /submit` is rate-limited with Flask-Limiter, `storage_uri="memory://"`
([`app.py`](app.py), limits in [`config.py`](config.py)):

```
10 per minute ; 100 per day
```

**Reasoning (not arbitrary):** a genuine creator writes before they submit, so even
heavy legitimate use — drafting, editing, resubmitting a revised piece — comfortably
fits inside 10/min; 100/day covers a prolific writer's whole day. An adversary
trying to *flood* the endpoint (to reverse-engineer the classifier by probing
thousands of variants, or to burn the shared Groq quota) is stopped at 10/min and
capped hard at 100/day. The limit is per client IP (`get_remote_address`).

**429 evidence** — the spec's 12-rapid-request loop against a running server (first
10 succeed, the rest are limited):

```
request 1  -> HTTP 200
request 2  -> HTTP 200
request 3  -> HTTP 200
request 4  -> HTTP 200
request 5  -> HTTP 200
request 6  -> HTTP 200
request 7  -> HTTP 200
request 8  -> HTTP 200
request 9  -> HTTP 200
request 10 -> HTTP 200
request 11 -> HTTP 429
request 12 -> HTTP 429
```

429 response body:

```json
{"error":"rate_limit_exceeded","limits":"10 per minute;100 per day","message":"Rate limit exceeded (10 per 1 minute). Slow down and retry later."}
```

`print()`. It records timestamp, `content_id`, `creator_id`, content type,
attribution, combined confidence, **all individual signal scores**, status, and any
appeal. `GET /log` returns recent entries as JSON.

## Stretch features

All four built and demonstrated.

1. **Ensemble detection (3+ signals + weighting).** The repetition/predictability
   signal ([`signals/ensemble.py`](signals/ensemble.py)) joins the LLM and
   stylometric signals under a documented weighted scheme (0.50 / 0.30 / 0.20,
   renormalised over the signals that ran). Visible in the ramen entry above, where
   all three contributed.
2. **Provenance certificate** ([`certificate.py`](certificate.py)). A creator
   *earns* a `✓ Verified Human` credential by submitting an original writing sample
   to `POST /certificate/verify`; if it reads convincingly human (combined
   confidence ≤ 0.40) and is ≥30 words, an HMAC-signed certificate is stored. After
   that, every `/submit` response and label for that creator carries the badge —
   independent of the per-post verdict. (Demo: `creator_bob` verified on a 66-word
   sample at confidence 0.18.)
3. **Analytics dashboard** ([`analytics.py`](analytics.py), `GET /analytics` JSON +
   `GET /dashboard` HTML). Detection-pattern distribution, **appeal rate** + review
   queue depth, and the chosen extra metric **signal-agreement rate** (how often the
   LLM and stylometric signals land in the same band — a low value flags ambiguous
   content or noisy signals), plus fallback-usage rate and verified-creator count.
4. **Multi-modal support.** A second content type, `content_type: "image_metadata"`,
   runs the pipeline over the **text that describes an image** (its
   description/caption/metadata) with an adapted LLM prompt, alongside ordinary
   `text`. To be precise about scope: this classifies the *attribution of the
   description text* (was the caption AI-written?) — it is the spec's "image descriptions or
   structured metadata" path, not binary image/vision analysis. (Demo: a flowery
   "ethereal… sublime essence" caption → `uncertain`, 0.68.)

## Known limitations

- **Formal, substantive human writing in a niche register** (legal briefs,
  academic abstracts, monetary-policy prose) is the content type most likely to be
  *over-flagged*. This is structural, not a data problem: Signal 2 keys on
  uniformity and formality, which such writing has by definition, so it pushes the
  score up. We mitigate it (the LLM's substance check + the disagreement→uncertain
  pull + the asymmetric AI bar usually land it in `uncertain`, as the monetary
  example shows), but a sufficiently dry, uniform human paragraph can still read as
  `uncertain` when a reader would call it obviously human. The appeal path exists
  precisely for this case.
- **Repetitive, simple-vocabulary human writing** (some poetry, song lyrics,
  children's stories) trips both the stylometric and repetition signals, which
  read low burstiness + high repetition as machine-like.
- **Short submissions (< ~40–50 words)** carry little structural signal; the
  stylometric score is shrunk toward neutral and the repetition signal is dropped,
  so very short text leans heavily on the single LLM signal.
- Perfect AI detection is unsolved; this system is calibrated to be *honest about
  uncertainty*, not to be right every time.
- **Read endpoints are intentionally open.** `GET /log`, `/analytics`, `/dashboard`,
  and `/reviewer/queue` have no authentication so the audit trail is easy to inspect
  for grading/demo. They expose decision metadata and short text excerpts — in a real
  deployment these would sit behind auth/RBAC (and the text excerpts would be
  redacted). The writeable endpoints (`/submit`, `/appeal`, `/certificate/verify`)
  are validated and rate-limited; only the read surface is unauthenticated by design.

## Spec reflection

- **Where the spec guided the implementation.** The hint "decide what 0.5 means to
  a *user* first, then build the math" directly shaped the design: I defined 0.5 as
  "we genuinely don't know," wrote the three label variants *before* any scoring
  code, then built `scoring.py` to produce a number whose bands map onto those
  labels. The asymmetric-error hint became the two concrete mechanisms in
  [Confidence scoring](#confidence-scoring).
- **Where the implementation diverged, and why.** The starter pairing implies
  combining signals by a fixed weighted average. My first build did exactly that —
  and it mislabeled the formal-human monetary paragraph as `likely_ai`, the very
  false positive the project warns about. So I diverged: scoring now *also* pulls
  the score toward 0.5 in proportion to how much the signals disagree, and it
  *drops* a signal that can't produce a reliable reading (the repetition signal on
  short text) instead of averaging in a misleading neutral. A plain weighted mean
  was not enough to honour the asymmetry the spec asked for.

## AI usage

I designed and built Provenance Guard. I used an AI coding assistant (Claude Code)
to help with some of the repetitive build work and to draft a few first cuts that
I then reviewed and reworked — the design decisions, calibration, and the parts
that actually matter were mine. Two specific instances:

1. **Calibrating the LLM signal's prompt — I directed it, then overrode it twice.**
   I had the assistant draft the Groq classification prompt and the deterministic
   fallback. Its first prompt was *under-confident* (it returned ~0.70 on
   obviously-AI text, so the clearly-AI paragraph wrongly landed `uncertain`). I
   added a 0–1 calibration rubric, which over-corrected and pushed the *formal
   human* monetary paragraph to ~0.82 (`likely_ai`), a false positive. I rewrote
   the rubric a third time to weigh **substance and specificity** over register
   ("a formal register is not evidence of AI; generic content-free scaffolding
   is") — the version in [`signals/llm_signal.py`](signals/llm_signal.py). The
   lesson was mine to find through testing: generic "be cautious"/"be decisive"
   instructions move *both* the AI and human cases; only the substance/specificity
   distinction separated them.
2. **Scaffolding it drafted, a scoring bug I caught.** I leaned on the assistant
   for repetitive boilerplate (the SQLite audit-log CRUD and some Flask
   route/error-handler plumbing), then reworked it. While checking the `/log`
   output during testing I noticed the combiner was averaging in every signal's
   neutral `0.5` even when a signal hadn't really run — biasing short submissions
   and recording a misleading `repetition_score: 0.5`. I changed the scoring to
   **drop** unavailable signals and renormalise, and to log `null` for any signal
   that didn't contribute (see [`scoring.py`](scoring.py) /
   [`pipeline.py`](pipeline.py)).

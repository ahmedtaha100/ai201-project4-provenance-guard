# Provenance Guard — Planning & Design Spec

> CodePath AI201 · Project 4. This document is the design contract: it was
> written before the implementation and is the single reference the code is built
> against. All numeric thresholds/weights quoted here live in [`config.py`](config.py),
> which is the runtime source of truth — if a number here ever disagrees with
> `config.py`, `config.py` wins and this doc is stale.

Provenance Guard is a Flask backend a creative-sharing platform plugs into to
answer one question about a submitted piece of writing: *was this written by a
human, or generated by an AI?* — and to answer it **honestly**, surfacing
uncertainty instead of forcing a binary, and giving creators a way to appeal.

---

## 1. Detection signals

The pipeline uses **three genuinely distinct signals** (two required + one for the
ensemble stretch). Each returns a normalised `ai_probability ∈ [0,1]` =
P(text is AI-generated), so they can be combined uniformly.

| # | Signal | Axis it measures | Output | Why it differs | Blind spot |
|---|--------|------------------|--------|----------------|------------|
| 1 | **LLM semantic** ([`signals/llm_signal.py`](signals/llm_signal.py)) — Groq `llama-3.3-70b-versatile` | **Meaning / voice** — does it *read* as AI when judged holistically (idea flow, hedging, generic "AI voice")? | `ai_probability` + indicators + reasoning | Only signal that understands *meaning*; can flag statistically-varied-but-templated text, or vouch for formal human prose. | Non-deterministic; easiest to fool with human-edited AI; needs a live API/key (deterministic fallback covers outages). |
| 2 | **Stylometric** ([`signals/stylometric.py`](signals/stylometric.py)) — pure Python | **Structure** — sentence-length *burstiness*, informality (contractions/punctuation/slang), lexical diversity (MATTR). | `ai_probability` from 3 weighted sub-metrics | Never reads meaning — only counts *shapes*; fails differently from the LLM, which is what makes the pair informative. | Formal human prose is uniform → false-positive; repetitive poems look AI; short text = unstable stats. |
| 3 | **Repetition / predictability** ([`signals/ensemble.py`](signals/ensemble.py)) — pure Python (stretch) | **Token-sequence predictability** — n-gram reuse + reliance on common, high-frequency words (a cheap perplexity proxy). | `ai_probability` from 2 weighted sub-metrics | A third orthogonal axis: not meaning, not sentence-shape, but *sequence repetition*. | Short texts lack n-grams; refrains/recipes repeat by design. |

**What each sub-metric measures (Signal 2):**
- *Burstiness* — coefficient of variation of sentence lengths. Humans alternate
  long and short; AI is metronomic. **Low CV → AI.** (weight 0.40)
- *Informality* — density of contractions, `! ? ...`, slang, lowercase sentence
  starts. Casual humans are rich in these; default AI prose is clean/formal.
  **High informality → human.** (weight 0.35)
- *Lexical diversity* — moving-average type-token ratio (MATTR, window 25),
  length-robust. **Very low diversity → mildly AI.** Weakest, noisiest sub-metric,
  hence the lowest weight (0.25).

**Combination → see §2.** Base pipeline combines Signals 1+2; the ensemble stretch
adds Signal 3 (weights renormalised over whichever signals ran).

## 2. Uncertainty representation

**`confidence` = P(content is AI-generated), 0..1.** We defined what the number
means to a *user* first, then built the math to hit it:

- **0.5 = "we genuinely don't know."** Signals are split or weak; the system
  refuses to attribute the work either way. This is the honest-uncertainty zone.
- Toward **1.0** = increasingly confident **AI**; toward **0.0** = increasingly
  confident **human**.

**Raw outputs → calibrated score** (in [`scoring.py`](scoring.py)):

1. **Weighted blend.** `raw = Σ wᵢ·pᵢ` over signals that ran. Base weights:
   LLM **0.60**, stylometric **0.40** (LLM is the more reliable single detector).
   Ensemble weights: **0.50 / 0.30 / 0.20**. Weights renormalise over present signals.
2. **Disagreement → uncertainty.** `spread = max(pᵢ) − min(pᵢ)`. The score is
   pulled toward 0.5 by `pull = 0.5 · spread`:
   `confidence = (1 − pull)·raw + pull·0.5`. Conflicting evidence reads as
   *uncertain*, never as a confident verdict. (This is what lands formal-human
   text — where the LLM says "human" but stylometry says "AI" — in *uncertain*.)
3. **Asymmetric banding** → the three thresholds below.

**Thresholds** (the bands separating likely-AI / uncertain / likely-human):

| Band | Condition | Meaning |
|------|-----------|---------|
| `likely_ai` | `confidence ≥ 0.70` | High bar — strong evidence required |
| `uncertain` | `0.40 < confidence < 0.70` | Wide middle, shifted toward the AI side |
| `likely_human` | `confidence ≤ 0.40` | Lower bar |

**Where the false-positive asymmetry lives (explicit):** the bar to assert **AI**
(distance 0.20 above 0.5) is deliberately larger than the bar to call **human**
(distance 0.10 below 0.5), and the uncertain band is shifted toward the AI side.
On a creative platform, wrongly branding a real person's writing "AI" is the
costlier error, so borderline-high scores fall into *uncertain* rather than being
asserted as AI. The disagreement-pull (step 2) reinforces this. A `0.51` →
*uncertain*; a `0.95` → *likely_ai* — meaningfully different labels, no binary flip.

**How we test it's meaningful:** [`tests/test_pipeline.py`](tests/test_pipeline.py)
asserts both-high → `likely_ai`, both-low → `likely_human`, split → `uncertain`
(pulled into 0.4–0.7), and that a 0.5-ish blend and a 0.95 blend get different
attributions and differ by >0.2 in confidence. The README shows two real
contrasting submissions with their actual numbers.

## 3. Transparency label design

Three variants, selected purely by the attribution band (so the label can never
disagree with the audit log). `{pct}` = `round(confidence·100)`. These are the
**canonical strings**, defined verbatim in [`labels.py`](labels.py) and quoted
verbatim in the README:

**`likely_ai` (high-confidence AI):**

```text
🤖 Likely AI-generated — Our automated check estimates a {pct}% likelihood that this text was produced mainly by an AI tool, which is above our high-confidence threshold. We're surfacing this for transparency. It is an estimate, not proof — if you wrote this yourself, you can appeal and a human will review it.
```

**`likely_human` (high-confidence human):**

```text
✍️ Likely human-written — Our automated check found writing patterns consistent with a human author (estimated {pct}% likelihood of AI generation — low). No strong AI-generation signals were detected. This is an automated estimate, not a guarantee of authorship.
```

**`uncertain`:**

```text
❓ Uncertain — We can't confidently tell whether this was written by a person or by an AI (estimated {pct}% likelihood of AI generation, inside our 'unsure' range). Our signals were weak or in conflict, so we are deliberately not labeling it either way. Treat it as unverified; the creator can add context through an appeal.
```

Design intent: plain language, the confidence is expressed as a percentage a
non-technical reader understands, and the AI/uncertain labels both name the
appeal path (the safety valve). The "uncertain" label explicitly says we are
*choosing not to guess* — honesty over false precision.

## 4. Appeals workflow

- **Who appeals:** the creator who submitted the content (or a moderator on their
  behalf), when they believe the attribution is wrong.
- **What they provide:** `POST /appeal` with `{ "content_id", "creator_reasoning" }`.
- **What the system does** ([`appeals.py`](appeals.py) → [`audit.py`](audit.py)):
  1. Look up the original decision by `content_id` (404 if unknown).
  2. Store `appeal_reasoning` + `appeal_timestamp` **beside** the original decision
     (the original attribution/confidence/signal scores are preserved).
  3. Flip `status` → `under_review`.
  4. Return a confirmation echoing the new status and the original verdict.
  - **No automated re-classification** — by design. The human reviewer is the
    safety valve for an unsolved problem.
- **What a reviewer sees** (`GET /reviewer/queue`): every `under_review` row with
  the original text excerpt, attribution, confidence, *both/all* individual signal
  scores, the creator's reasoning, and timestamps — enough to make a human call.

## 5. Anticipated edge cases (specific)

1. **Formal human writing flagged as AI (the core false-positive).** The
   monetary-policy paragraph is uniform, complex, and contraction-free, so the
   stylometric signal scores it high (~AI). *Mitigation:* the LLM signal should
   read it as human; the resulting disagreement pulls the score toward *uncertain*
   (step 2 of §2), and the asymmetric AI bar (0.70) keeps it out of `likely_ai`.
2. **Repetitive, simple-vocabulary poem misread as AI.** Short lines, low lexical
   diversity, low burstiness → stylometric + repetition both lean AI. *Mitigation:*
   LLM context + the appeal path; documented as a known limitation in the README.
3. **Lightly human-edited AI text (the false-negative).** The remote-work paragraph
   is AI-shaped but lightly humanised; signals land mid-range → *uncertain*, which
   is the correct, honest answer rather than a false "human" clear.
4. **Very short input** ("Nice work!"). Sentence/n-gram statistics are meaningless;
   the stylometric and repetition signals shrink toward 0.5 below
   `MIN_RELIABLE_WORDS` (40) so they cannot dominate a thin sample.
5. **Empty / non-string / oversized input.** Rejected with HTTP 400 before any
   signal runs (`MAX_TEXT_CHARS` = 50 000).

## Architecture

```
                       Provenance Guard — submission flow
  ┌──────────────────────────────────────────────────────────────────────────┐
  │ POST /submit  { text, creator_id, content_type? }                         │
  └───────────────┬──────────────────────────────────────────────────────────┘
                  │ raw text (validated: non-empty, ≤50k chars, known type)
                  ▼
        ┌───────────────────┐   ┌────────────────────┐   ┌───────────────────┐
        │ Signal 1: LLM      │   │ Signal 2:           │   │ Signal 3:          │
        │ (Groq semantic,    │   │ stylometric         │   │ repetition/predict │
        │  + fallback)       │   │ (structural)        │   │ (ensemble stretch) │
        └─────────┬──────────┘   └─────────┬───────────┘   └─────────┬─────────┘
            p_llm │ 0..1            p_style │ 0..1           p_rep    │ 0..1
                  └────────────┬────────────┴───────────────┬────────┘
                               ▼ (weighted blend + disagreement→0.5 pull)
                       ┌───────────────────┐
                       │ scoring.combine    │ → confidence ∈ [0,1] (P(AI))
                       │ + asymmetric bands │ → attribution (likely_ai/uncertain/likely_human)
                       └─────────┬─────────┘
                                 ▼ confidence + attribution
                       ┌───────────────────┐        ┌───────────────────────┐
                       │ labels.make_label  │        │ certificate (stretch): │
                       │ → 1 of 3 variants  │◀───────│ verified-human badge?  │
                       └─────────┬─────────┘        └───────────────────────┘
                                 ▼ content_id + attribution + confidence + label + signal scores
                       ┌───────────────────┐
                       │ audit.record (SQLite) │  ← timestamp, content_id, creator_id,
                       └─────────┬─────────┘       attribution, confidence, ALL signal scores, status
                                 ▼
                    JSON response (content_id, attribution, confidence, label, signals)

                          Appeal flow
  POST /appeal {content_id, creator_reasoning}
        │
        ▼ look up content_id ──(not found)──▶ 404
        │ (found)
        ▼ audit.record_appeal: status → "under_review", store reasoning + timestamp
        ▼
  JSON confirmation  ───────▶  GET /reviewer/queue surfaces it to a human
```

**Submission narrative.** A `POST /submit` is validated, then the raw text fans
out to the LLM, stylometric, and repetition signals in parallel concept; their
three `ai_probability` scores are blended, pulled toward 0.5 in proportion to how
much they disagree, and banded by asymmetric thresholds into a `confidence` +
`attribution`. That maps to one of three transparency labels, the whole decision
(including every individual signal score) is written to the SQLite audit log under
a unique `content_id`, and the response returns the id, attribution, confidence,
and label.

**Appeal narrative.** A `POST /appeal` carrying that `content_id` looks up the
original decision, attaches the creator's reasoning, flips the status to
`under_review`, and returns a confirmation — no re-classification — so a human
reviewer can later see the full original verdict alongside the appeal.

## AI Tool Plan (Milestones 3–5)

This spec + the diagram are the context I feed to the AI coding assistant when
generating each milestone's first cut, and the checklist I use to verify and
correct its output before moving on.

**M3 — submission endpoint + Signal 1.**
- *Provide:* §1 (Signal 1 row), the Architecture diagram, the response contract.
- *Generate:* Flask skeleton + `POST /submit` (returns `content_id`, attribution,
  placeholder confidence/label), `signals/llm_signal.py` (Groq + deterministic
  fallback returning a structured `SignalResult`), SQLite audit log + `GET /log`.
- *Verify:* call `llm_signal.assess()` directly on a few inputs and confirm the
  structured output; `curl POST /submit` shows `content_id`/`attribution` in JSON
  and a row in `/log`. **Verify before moving on.**

**M4 — Signal 2 + confidence scoring.**
- *Provide:* §1 (Signal 2), §2 (uncertainty), the diagram.
- *Generate:* `signals/stylometric.py` (2–3 metrics → one score), `scoring.py`
  combining signals per §2.
- *Verify:* run the 4 starter inputs; confirm clearly-AI ≫ clearly-human and the
  scoring **matches the §2 thresholds** (AI tools often drift from the documented
  ranges — check and correct). Extend the audit log with both signal scores.
  **Verify before moving on.**

**M5 — production layer + stretch.**
- *Provide:* §3 (labels), §4 (appeals), the diagram.
- *Generate:* `labels.py` (3 variants, exact §3 text), `POST /appeal`, Flask-Limiter
  on `/submit`, full audit fields; stretch: `ensemble.py`, `certificate.py`,
  `analytics.py`, multi-modal `content_type`.
- *Verify:* all three labels reachable; `/appeal` sets `under_review` and logs
  reasoning (check via `/log`); 12-request loop shows `200×10` then `429`; ≥3
  structured audit entries incl. an appeal. **Verify**, then a final whole-repo
  review + secret-handling check.
